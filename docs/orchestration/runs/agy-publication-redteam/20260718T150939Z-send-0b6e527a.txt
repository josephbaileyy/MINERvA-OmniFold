### 2. Task-Class Routing Table
*   **Login-Safe**: Verifiers, pure Python metadata scripts, code-generation turns, manifest builders, and orchestration/git ops.
*   **Held Interactive (`alloc_run.sh` / `urgent`)**: Fast-turnaround single-node compute (<3h) where latency matters (e.g., covariance component building, 5D→4D projections, final atomic adoption math).
*   **Shared/Regular Batch (`shared` / `regular`)**: CPU-heavy arrays and tasks exceeding 3h or tolerant of fairshare queuing (e.g., event loops, 10-endpoint unfolds, large hadd tasks).
*   **GPU Batch (`urgent_gpu` / `shared_gpu`)**: PET model training and evaluation (e.g., F2/F3 GPU jobs, F7 nominals).
*   **Array**: Fully independent playlist or endpoint processing (e.g., 12-playlist event loops).
*   **First-Start-Wins Hedge**: A simultaneous submission of a batch job (for long-tail queue safety) and an interactive allocation request (for latency).

### 3. Exact Conditions for a Double-Write-Free Hedge
To safely hedge batch vs. interactive without corrupting data, the script MUST implement:
1.  **Shared Lock**: `flock -n 9 /path/to/lockfile || exit 0` to ensure only one instance executes the block.
2.  **Collision-Free Staging**: Output to a strictly unique temporary path (`output.tmp.$SLURM_JOB_ID` or `$RANDOM`).
3.  **Winner Receipt**: An atomic `mv` to the final destination upon successful completion.
4.  **Loser Cancellation / Skip-if-Exists**: The script must check `if [[ -f final_output.json ]]; then exit 0; fi` before attempting the lock.
5.  **Ownership**: Initiated by the same owning agent/orchestrator so namespaces remain intact.
