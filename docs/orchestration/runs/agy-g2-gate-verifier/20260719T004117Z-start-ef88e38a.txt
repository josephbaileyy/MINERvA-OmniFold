### 2. Blocking Defect
**Defect:** The content-validated resume logic fails to fail-closed on one-sided published states (ROOT-only or receipt-only). Currently, if only one of the artifacts exists (e.g., from a failure after the ROOT rename but before the receipt is written), the check `[[ -s ... && -s ... ]]` evaluates to false. The script then bypasses the resume guard, runs the event loop, and executes `mv -f`, which silently overwrites and destroys the prior published evidence.

- **File:** `nd-unfolding/pet/sbatch_g2_fullevent_evloop_array.sh`
- **Line:** 91
- **Current:** `if [[ -s "${FINAL_ROOT}" && -s "${RECEIPT}" ]]; then`
- **Minimal Correction:** `if [[ -e "${FINAL_ROOT}" || -e "${RECEIPT}" ]]; then`

*Why this works:* Changing the condition to `||` and using `-e` forces any existing state (stale, partial, or one-sided) into the evaluation block. If one file is missing, the Python resume script will crash (throwing an exception when trying to open the missing file) or exit non-zero, correctly dropping into the `else` block and `die`-ing to protect the existing evidence.

### 3. Nonblocking Improvements
- **Manifest Content Drift Guard:** The array currently checks only that `DATA_MANIFEST` and `MC_MANIFEST` exist and are non-empty (lines 71-72). If manifest content drifts, the script will silently use the drifted manifests and simply record their new hashes. - **Defense-in-Depth for Atomic Publication:** Change `mv -f` (line 142) to `mv -n` (no-clobber). This adds a filesystem-level safety net that guarantees an atomic publication will never overwrite an existing file, even if the resume guard logic is inadvertently bypassed.
